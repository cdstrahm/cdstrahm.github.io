Betology Analytics System Overview

Core Objective & Overview

When designing the analytical pipeline for Betology, my primary goal was to mimic a small quantitative trading system. Additionally, I wanted the pipeline to be automated, allowing me to futher develop other aspects of Betology or improve the system, rather than manually operating it daily.

Primary functions of the pipeline:

  • Ingest and store raw sports data
  • Engineer predictive features
  • Train and evaluate predictive models
  • Generate and distribute daily predictions
  • Track historical performance

While I plan to continue developing, optimizing, and improving the system, I am proud of its current state and daily performance. The projects involved in developing the pipeline have taught me how to design systems for accuracy, reliability, and long-term maintainability. These projects have allowed me to gain experience working with and handling real, imperfect data, and utilizing it in a predictive context.

Infrastructure & Execution Environment

This layer ensures storage, secure access, and unattended execution across environments.

Technologies used:

  • Linux VPS (Ubuntu) - Hosts the production database
  • MySQL - Centralized the relational database
  • SSH - Secure server access
  • Environmental variables/secrets - Credentials injected at runtime rather than hardcoded
  • GitHub Actions - Headless execution environment for scheduled workflows

The system is designed to operate independently of any single machine. Local development and production execution share the same logic, ensuring consistency across environments.

Data Ingestion & Storage

This part of the system is designed to ingest and normalize raw API data, making it queryable. It also utilizes defensive handling of missing games and team-name mismatches. When implementing the ETL pipeline, it was assumed that external data is unreliable and must be verified at each step.

Technologies used:

  • Python - Scripts developed in Python
  • REST APIs - CollegeBasketballData and KenPom's APIs used to acquire data
  • Requests - Library used to interact with external APIs
  • Pandas - Used to data manipulation
  • SQL - Data managed by RDMS to support SQL querying
  • SQLAlchemy - Used to interact with SQL database in Python

Design characteristics:

  • Incremental updates instead of full rebuilds
  • Explicit date indexing to preserve temporal integrity
  • Defensive handling of missing games and naming mismatches
  • Normalized schema supporting foreign key integrity

This layer supports downstream feature engineering, historical backtesting, and model evaluation.

Feature Engineering Pipeline

The feature engineering pipeline is performed entirely in code and uses only information available prior to the prediction date. The design prevents temporal leakage and ensures reproducibility.

Features include:

  • Rolling averages over prior games
  • Lagged performance metrics
  • Team-to-team differentials
  • Home/away indicators
  • Pace and efficiency interactions

Games with incomplete feature sets are explicitly logged and excluded from prediction rather than being silently imputed. The core design principles used prioritize foreign key relationships for integrity and tables structured to support rolling-window queries.

Modeling & Training

Multiple model types are used to balance interpretability and predictive performance. Logistic regression models serve as baselines, while gradient-boosted decision trees (XGBoost) capture non-linear relationships. Predictions are probabilistic rather than categorical, enabling confidence-based filtering and evaluation.

Technologies used:

  • scikit-learn - Machine learning library used for regression
  • XGBoost - Machine learning library used for gradient-boosted tree modeling
  • pandas / NumPy - Feature matrix preparation

Evaluation emphasizes accuracy and AUC. Modeling logic is fully decoupled from data ingestion and distribution layers.

Prediction Generation

This layer is responsible for transforming engineered features and trained models into structured, reproducible outputs and delivering them through automated channels.

Daily prediction workflows follow a deterministic execution path:

  1. Load upcoming games from the database
  2. Join the most recent valid feature set
  3. Align feature schema with the trained model
  4. Validate input completeness
  5. Generate probabilistic win predictions

Technologies used:

  • Python - Core execution logic
  • pandas - Feature alignment and output formatting
  • NumPy - Array transformations

Distribution

Once predictions are generated, they are formatted and distributed programmatically. Distribution is downstream-only and cannot modify or influence prediction outputs.

Technologies used:

  • Discord Webhooks - Automated message and file delivery
  • SMTP (smtplib) - Email distribution
  • CSV / structured file exports - Archivable output format
  • Requests - Webhook communication

Each distribution channel operates independently, allowing the system to scale to additional endpoints without altering upstream modeling logic.
